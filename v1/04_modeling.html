Antes de entrenar ningún modelo se establece el MAE de referencia usando predictores puramente heurísticos. Cualquier modelo aprendido debe superar ambos baselines para justificar su complejidad adicional.

3 · Entrenamiento LightGBM¶

Se usa LightGBM con los siguientes hiperparámetros de partida, elegidos para un primer modelo razonablemente bueno sin tuning exhaustivo:

  • num_leaves=127 — árbol moderadamente complejo sin sobreajuste
  • learning_rate=0.05 — conservador para que el early stopping encuentre el óptimo
  • min_child_samples=50 — regularización mínima por hoja
  • early_stopping_rounds=50 — para en validación si no mejora en 50 rondas
In [30]:
# Construir datasets LightGBM con soporte para categoricals
cat_cols_idx = [FEATURES.index(c) for c in CAT_FEATURES if c in FEATURES]
cat_itinerari_idx = [FEATURES.index('idItinerari')] if 'idItinerari' in FEATURES else []
all_cat_idx = cat_cols_idx + cat_itinerari_idx

dtrain = lgb.Dataset(X_train, label=y_train,
                     categorical_feature=all_cat_idx, free_raw_data=False)
dval   = lgb.Dataset(X_val,   label=y_val,
                     categorical_feature=all_cat_idx, free_raw_data=False,
                     reference=dtrain)

params = {
    'objective':         'regression',
    'metric':            'mae',
    'num_leaves':        127,
    'learning_rate':     0.05,
    'feature_fraction':  0.8,
    'bagging_fraction':  0.8,
    'bagging_freq':      5,
    'min_child_samples': 50,
    'lambda_l1':         0.1,
    'lambda_l2':         0.1,
    'verbose':           -1,
    'n_jobs':            -1,
    'seed':              42,
}

callbacks = [
    lgb.early_stopping(stopping_rounds=50, verbose=True),
    lgb.log_evaluation(period=100),
]

print('Entrenando LightGBM...')
model = lgb.train(
    params,
    dtrain,
    num_boost_round=2000,
    valid_sets=[dval],
    callbacks=callbacks,
)

print(f'\nMejor iteración: {model.best_iteration}')
print(f'MAE validación:  {model.best_score["valid_0"]["l1"]:.4f}')

def evaluar(y_true, y_pred, nombre):
    mae  = mean_absolute_error(y_true, y_pred)
    rmse = mean_squared_error(y_true, y_pred) ** 0.5
    r2   = r2_score(y_true, y_pred)
    return {'split': nombre, 'MAE': mae, 'RMSE': rmse, 'R2': r2, 'n': len(y_true)}

resultados = []
for X, y, nombre in [(X_train, y_train, 'Train'),
                     (X_val,   y_val,   'Val'),
                     (X_test,  y_test,  'Test')]:
    pred = model.predict(X, num_iteration=model.best_iteration)
    resultados.append(evaluar(y, pred, nombre))

res_df = pd.DataFrame(resultados).set_index('split')

Lectura por indicador:

  • R² = 0.951 en test: el modelo explica el 95.1% de la varianza del factor de tráfico. Es un valor muy alto — indica que los lags, rolling windows y features temporales capturan casi toda la dinámica habitual del tráfico.
  • Degradación Train→Test controlada: el MAE sube de 0.0028 a 0.0056 (×2), lo que es esperable y no indica sobreajuste grave. El R² cae solo 1.6 puntos (0.967→0.951), confirmando buena generalización.

El R² de test (0.951) es incluso ligeramente mejor que el de validación (0.949), lo que sugiere que 2024 tiene un comportamiento más regular o predecible.

El MAE de 0.0056 sobre un factor normalizado (1.0 = nivel de referencia) equivale a un error medio de ±0.56% del nivel de referencia — prácticamente insignificante en condiciones normales. El modelo falla principalmente en los picos extremos (factor > 1.5) asociados a eventos no recurrentes, que es precisamente el gap que justifica el objetivo del TFM.

5 · Importancia de features — SHAP¶

SHAP (SHapley Additive exPlanations) descompone cada predicción en la contribución marginal de cada feature. A diferencia del feature importance por ganancia de LightGBM, SHAP es consistente y comparable entre features.

Beeswarm plot SHAP (top 20 features):

Las features con mayor impacto son, en orden:

  • factor_min, factor_max — los límites del factor en la misma hora son muy informativos: si el mínimo ya es alto, el medio también lo será.
  • tiempo_actual — el tiempo de viaje real contiene información complementaria al factor.
  • factor_std — la volatilidad intra-horaria contextualiza el nivel medio; más dispersión implica más incertidumbre.
  • tendencia_media — la dirección del tráfico (cargando/descargando) es predictiva del estado siguiente.
  • Los lags (tiempo_lag_168h, factor_lag_24h/168h) aparecen con efecto positivo: un valor alto de lag → predicción más alta.
  • idTram — el identificador captura diferencias estructurales entre vías que el modelo aprende como categoría.
  • hora_dia — efecto no lineal con picos en mañana y tarde.

Nota relevante: los lags cortos (1h, 2h, 3h) no aparecen en el top 20 pese a su alta correlación de Pearson. En presencia de factor_min, factor_max y factor_std del mismo timestamp, el modelo no necesita el lag 1h tanto como se esperaba — estas tres variables ya capturan el estado actual del tramo con más riqueza que el valor de hace una hora.

7.2 Hallazgos clave¶

Sobre la autocorrelación temporal Las features derivadas (SHAP 0.0043) superan a los lags (0.0035) como bloque más importante. delta_tiempo, ratio_tiempo y factor_rango_hora — estadísticos de la misma hora predicha — resultan más informativos que los valores del pasado. Junto con los lags, concentran el 60 % de la importancia SHAP total.

Sobre la fiabilidad del sensor El bloque Sensor (sensor_trafico_ok, SHAP 0.0008) supera a Meteo (0.0004), confirmando que el flag de fiabilidad generado por Isolation Forest y peer comparison aporta información útil al modelo: las horas con lecturas anómalas tienen un patrón diferente que LightGBM aprende a gestionar.

Sobre los patrones temporales El error varía fuertemente según la franja horaria: la tarde concentra el mayor error (MAE = 0.0092) y la noche el menor (0.0015). En hora punta el MAE es 2.2× mayor que fuera de ella (0.0092 vs 0.0041), precisamente cuando la predicción es más valiosa para la gestión del tráfico.

Sobre el impacto de los eventos El análisis festival vs. sin festival muestra que el modelo no empeora en presencia de eventos — el MAE con festival (0.0045) es incluso mejor que sin festival (0.0057). Sin embargo, el bloque de eventos tiene la importancia SHAP más baja (0.0001), lo que indica que las features de evento actuales no añaden información diferencial más allá de lo que capturan los lags y las features temporales.

Sobre la variabilidad espacial La distribución del MAE por corredor es extremadamente asimétrica: 9 de 12 corredores tienen MAE ≈ 0, mientras que los corredores 11 (MAE = 0.0493) y 14 (MAE = 0.0369) concentran la mayor parte del error global. Estos dos corredores merecen investigación específica para determinar si coinciden con zonas de alta actividad de eventos en Barcelona.
